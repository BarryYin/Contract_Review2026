# ContractAI — AI 工具使用说明

> 本文档详细说明 ContractAI 智能合同合规审查平台中 AI 工具的使用方式、应用环节及设计思路。

---

## 一、AI 工具列表

| 工具 | 提供方 | 模型标识 | 用途 |
|------|--------|----------|------|
| **Step 3.5 Flash** | 阶跃星辰 (StepFun) | `step-3.5-flash` | 文本理解、结构化提取、NER、合规分析、双语对比、风险识别 |
| **Step step-1v-8k** | 阶跃星辰 (StepFun) | `step-1v-8k` | 视觉模型，用于图片/扫描件 OCR 文字识别 |
| **Claude / Hermes Agent** | Anthropic / Nous Research | — | 开发辅助：代码生成、架构设计、文档撰写 |

所有 LLM 调用统一通过 StepFun OpenAI 兼容 API（`https://api.stepfun.com/v1`）完成，使用 `httpx` 异步 HTTP 客户端，`temperature=0.1` 以确保输出稳定。

## 二、各环节 AI 应用详解

### 2.1 文档解析

**涉及文件**: `services/doc_parser.py`、`services/ocr_service.py`

| 输入 | 处理方式 | AI 工具 |
|------|----------|---------|
| DOCX | `python-docx` 提取段落文本 + 表格内容 | 无（纯程序化） |
| PDF（文字版） | `PyPDF2` 提取嵌入文本 | 无（纯程序化） |
| PDF（扫描件） | `pdf2image` 转图片 → 逐页 OCR | **Step step-1v-8k** |
| 图片（JPG/PNG 等） | Base64 编码后发送至视觉模型 | **Step step-1v-8k** |

OCR Prompt 策略：
- 语言自动检测（支持中/英/混合）
- 要求模型保持原始格式和排版，包括标题、段落、编号、表格
- 对模糊和手写文字做最大努力辨认
- 明确禁止模型添加解释性文字或 markdown 标记

解析后文本按正则模式拆分为条款列表（支持中英文编号格式），供下游分析使用。

### 2.2 结构化信息提取

**涉及文件**: `services/structured_parser.py`

调用 **Step 3.5 Flash**，使用严格的 JSON Schema 约束 Prompt，从合同全文提取以下字段：

- `contract_type`：合同类型（7 类：采购/服务/保密/劳动/租赁/技术/其他）
- `parties`：当事人信息（名称、地址、法定代表人、角色）
- `contract_period`：合同期限（起止日期 + 文字描述）
- `payment_terms`：付款条款（金额、币种、周期、方式）
- `breach_liability`：违约责任
- `dispute_resolution`：争议解决（仲裁机构/管辖法院）
- `confidentiality`：保密条款
- `intellectual_property`：知识产权条款
- `termination`：终止条款

技术保障：
- Prompt 中给出完整的 JSON Schema 示例，约束模型输出格式
- `temperature=0.1`，`max_tokens=16384`
- 健壮 JSON 解析：支持 markdown 代码块清理、正则提取、截断补全等多重容错
- 字段规范化层（`_normalize_structured`）：确保所有字段存在且类型正确
- LLM 不可用时返回空 fallback 结果，不影响系统运行

### 2.3 NER 命名实体识别

**涉及文件**: `services/ner_service.py`

调用 **Step 3.5 Flash**，从合同文本中提取 6 类命名实体：

| 实体类型 | 说明 | 示例 |
|----------|------|------|
| `DATE` | 日期和时间 | "2024年1月1日"、"签订之日起30天" |
| `MONEY` | 金额 | "人民币壹佰万元整"、"$50,000" |
| `LOCATION` | 地点 | "北京市海淀区" |
| `COMPANY` | 公司/组织名称 | "北京某某科技有限公司" |
| `PERSON` | 人名 | "张三"、"John Smith" |
| `CONTRACT_ID` | 合同编号 | "HT-2024-001" |

每个实体附带 `context`（上下文约 50 字符）和 `position_hint`（位置提示），支持前端高亮定位。输出结果自动过滤无效实体，按类型统计数量。

### 2.4 双语一致性分析

**涉及文件**: `services/bilingual_analyzer.py`

针对中英双语合同，调用 **Step 3.5 Flash** 执行双轨对比：

1. **语言分割**：基于 CJK 字符比例的启发式算法，将全文按段落分为中文/英文两部分
2. **并行提取**：对中文部分和英文部分分别调用 `extract_structured_info()`，获取各自的结构化信息
3. **字段级对比**：逐字段比较双方提取结果，标记为 `consistent`（一致）/ `inconsistent`（不一致）/ `missing`（缺失）
4. **一致性评分**：根据不一致项数量计算 0-100 分（不一致 -15 分/项，缺失 -8 分/项）

对比维度涵盖：合同类型、当事人、合同期限、付款条款、违约责任、争议解决、保密条款、知识产权、终止条款。

### 2.5 合规规则引擎

**涉及文件**: `services/rule_engine.py`、`core/rules.yaml`

基于 YAML 配置的预置规则库（7 条规则），通过**关键词匹配 + 正则表达式**进行确定性检查：

| # | 规则名称 | 严重程度 | 检查方式 |
|---|----------|----------|----------|
| 1 | 违约金比例上限 | high | 关键词 + 正则（`违约[金罚].*?(\d{1,3})[%％]`） |
| 2 | 付款周期超过60天 | medium | 关键词 + 正则（`(?:付款|支付|结算).{0,20}?(\d{2,3})\s*[天日]`） |
| 3 | 保密期限缺失或过短 | medium | 关键词 + 正则（`保密.{0,30}?(\d{1,2})\s*[年月]`） |
| 4 | 争议解决条款不明确 | high | 关键词 + 正则 |
| 5 | 知识产权工作成果范围不清晰 | high | 关键词 + 正则 |
| 6 | 免责条款单方面不对等保护 | high | 关键词 + 正则 |
| 7 | 终止条款赋予一方无条件单方解除权 | high | 关键词 + 正则 |

每条规则均配有：
- `legal_basis`：引用具体法条（如《民法典》第585条）
- `risk_description_template`：风险描述模板
- `check_config.semantic`：语义检查指引（标记待 LLM 深度分析）
- `contract_types`：适用的合同类型范围

规则引擎为纯确定性执行（无 AI 参与），保证同一输入始终产生相同结果，可解释性强。

### 2.6 语义风险识别

**涉及文件**: `services/compliance_engine.py`

调用 **Step 3.5 Flash** 进行深度语义分析，输出**三要素格式**的风险报告：

```
每个 Issue 包含三个核心字段：
├── risk_description（风险描述）：用非法律专业语言说明风险及后果（50-200 字）
├── legal_basis（法律依据）：引用具体法条，如"依据《民法典》第577条"
└── modification_example（修改示例）：
    ├── original：合同原文中有问题的段落（完整摘录）
    └── suggested：建议修改后的完整文本（可直接替换）
```

系统 Prompt 中明确了 8 大审查要点：付款条件、知识产权归属、保密与竞业限制、争议解决、终止条件、免责条款、禁止条款、权利义务对等性。

### 2.7 评分计算

**涉及文件**: `services/scorer.py`

采用**规则引擎命中 + LLM 风险维度**的双轨加权评分：

**7 大评分维度及权重：**

| 维度 | 权重 |
|------|------|
| 违约责任 | 20% |
| 付款条款 | 15% |
| 保密义务 | 15% |
| 知识产权 | 15% |
| 免责条款 | 15% |
| 争议解决 | 10% |
| 终止条款 | 10% |

**计算流程：**
1. 规则命中的 RuleHit 按名称映射到维度
2. LLM 的 issue 按标题/描述关键词启发式映射到维度
3. 每维度扣分：high = -25 分、medium = -15 分、low = -5 分
4. 各维度得分 = max(0, 100 - 累计扣分)
5. 加权总分 = Σ(维度分 × 权重) / Σ(权重)
6. 风险等级判定：≥3 个 high → high；≥1 个 high 或 ≥5 个 medium → medium；否则 low

## 三、Prompt 设计思路摘要

### 3.1 核心策略

| 策略 | 实现 |
|------|------|
| **JSON Schema 约束** | 在 System Prompt 中给出完整 JSON 结构示例和字段说明，要求"只输出 JSON，不要输出其他内容" |
| **低温度采样** | 统一使用 `temperature=0.1`，最大化输出确定性和可重复性 |
| **分步角色扮演** | 每个 Prompt 赋予明确角色（如"合同信息提取专家"、"合规审查专家"、"NER 专家"） |
| **字段级校验** | LLM 返回后经过规范化层（normalize），确保字段存在、类型正确、缺失值填充 |
| **多重容错解析** | 5 层 JSON 解析策略：直接解析 → markdown 清理 → 正则提取 → 截断补全 → 全文兜底 |

### 3.2 三要素输出约束

合规分析 Prompt 中明确要求每个 Issue 同时包含三个字段，并在 System Prompt 中反复强调：
- "必须引用真实存在的具体法条"
- "绝对禁止编造法条编号"
- "宁可只引用模糊依据也不能编造法条编号"
- "modification_example 的 suggested 应是可直接替换的修改版本"

### 3.3 OCR Prompt 策略

- 语言提示前置（"这是一份中文/英文文档"）
- 要求"保持原始格式和排版"
- 要求"模糊或手写文字尽量辨认"
- 禁止添加解释或 markdown 标记

## 四、注意事项

### 4.1 法条引用准确性保证

- **Prompt 级约束**：System Prompt 中明确要求"必须引用真实法条"、"禁止编造法条编号"
- **兜底策略**：当 LLM 不确定时，要求"宁可只引用模糊依据也不能编造法条编号"
- **规则引擎互补**：7 条预置规则中的 `legal_basis` 字段为人工编写、经过验证的真实法条引用，与 LLM 输出形成交叉验证
- **人工复核建议**：系统输出的法律建议仅供参考，不构成法律意见，最终审查结果应由专业法律人员复核

### 4.2 幻觉防范措施

| 措施 | 说明 |
|------|------|
| **低温度采样** | `temperature=0.1`，抑制模型的随机生成倾向 |
| **JSON Schema 强约束** | 严格限定输出结构，减少自由生成空间 |
| **原文摘录要求** | `modification_example.original` 要求完整摘录合同原文，避免凭空编造 |
| **Fallback 机制** | 所有 LLM 服务均有 fallback：API 不可用时返回空结果，不影响系统运行 |
| **字段规范化** | 对 LLM 输出做类型校验和缺失值填充，过滤无效数据 |
| **多源交叉验证** | 结构化提取（LLM）与规则引擎（确定性）结果并存，可交叉比对 |

### 4.3 数据安全承诺

- **文件哈希校验**：上传文件计算 SHA-256 哈希值，记录在审计日志中，确保文件完整性可追溯
- **审计日志**：所有关键操作（上传、审查、采纳/拒绝、导出）均记录审计轨迹
- **本地存储**：合同文件和审查结果存储在本地服务器，不向第三方传输原始文件
- **API 密钥管理**：LLM API 密钥通过环境变量注入，不硬编码在代码中（生产环境建议使用密钥管理服务）
- **传输安全**：与 StepFun API 的通信使用 HTTPS 加密传输

## 五、完整审查流程（AI 工具调用链）

```
用户上传合同文件
    │
    ▼
[1] 智能文档解析 (doc_parser.py)
    ├── DOCX → python-docx 提取文本（无 AI）
    ├── PDF 文字版 → PyPDF2 提取文本（无 AI）
    ├── PDF 扫描件 → Step step-1v-8k OCR ★
    └── 图片 → Step step-1v-8k OCR ★
    │
    ▼
[2] 并行执行 (asyncio.gather)
    ├── 结构化提取 → Step 3.5 Flash ★
    ├── NER 实体识别 → Step 3.5 Flash ★
    └── 双语一致性分析（仅双语合同）→ Step 3.5 Flash ★×2
    │
    ▼
[3] LLM 合规分析 → Step 3.5 Flash ★
    │   输出三要素：风险描述 + 法律依据 + 修改示例
    │
    ▼
[4] 规则引擎评估 (rule_engine.py)
    │   7 条 YAML 预置规则，关键词+正则匹配（无 AI）
    │
    ▼
[5] 多维度评分 (scorer.py)
    │   规则命中 + LLM 问题 → 加权评分（无 AI）
    │
    ▼
[6] 审计日志 + 结果存储 (audit_service.py)
    │   SHA-256 文件哈希 + 操作记录
    │
    ▼
输出完整审查报告 JSON
```
